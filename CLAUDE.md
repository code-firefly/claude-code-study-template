# Claude Code 学习计划 - AI 工作流程指南

## 项目概述

这是一个 **Claude Code 学习计划**项目，用于系统化学习 Claude Code 的使用方法、MCP 协议和 Agent 开发。

### 学习目标
- 掌握 Claude Code 的核心功能和配置
- 学习 MCP (Model Context Protocol) 协议
- 了解 Agent SDK 开发
- 积累实战项目经验

### 预计学习时长
- **最小可行路径**（P0 模块）：约 1 周
- **核心能力路径**（P0 + P1）：约 6-8 周
- **全栈学习路径**（全部模块）：约 10-14 周

## 学习工作流程命令

当用户请求以下操作时，按照对应模式执行。**支持自然语言触发**。

### 开始学习

**触发方式**（任意一种）：
- "开始学习 `<模块名>`"
- "启动 `<模块名>` 的学习"
- "开始学习模块 `<模块名>`"

**执行步骤**：

0. **检查知识缓存**：
   - 读取模块的 `knowledge/.metadata.json`（如果存在）
   - **情况 A：缓存存在**
     - 显示：`发现本地缓存 (缓存日期: YYYY-MM-DD)`
     - 询问用户：
       - 1. 使用本地缓存（快速开始）
       - 2. 更新缓存（拉取最新资料）
   - **情况 B：缓存不存在**
     - 显示：`未找到知识缓存`
     - 询问用户：
       - 1. 初始化缓存（推荐，首次拉取）
       - 2. 跳过缓存（直接开始学习）

1. 查找模块路径（从以下映射）：
   - `claude-code-core` → `01-基础入门/claude-code-core`
   - `mcp-basics` → `01-基础入门/mcp-basics`
   - `agent-sdk` → `02-进阶探索/agent-sdk`
   - `mcp-advanced` → `02-进阶探索/mcp-advanced`
   - `openclaw-ecosystem` → `02-进阶探索/openclaw-ecosystem`
   - `everything-claude-code` → `02-进阶探索/everything-claude-code`
   - `cc-switch` → `03-实战应用/cc-switch`
   - `spec-kit` → `03-实战应用/spec-kit`
   - `projects` → `03-实战应用/projects`

2. 验证模块是否已开始学习（检查 checklist.md 的学习模式选择）
   - 如果已开始，显示警告信息

3. 询问用户选择学习模式：
   - 1. 📚 快速模式（约 3 天）- 理论为主
   - 2. 🛠️ 完整模式（约 1 周）- 理论+实践

4. 显示更新预览，包括：
   - 模块信息（名称、优先级、模式、开始日期）
   - 将要更新的文件列表
   - 等待用户确认

5. 用户确认后，更新以下文件：
   - **checklist.md**：勾选对应的学习模式 checkbox
   - **notes.md**：设置学习模式和开始日期
   - **PROGRESS.md**：更新模块状态为"进行中"，添加学习日志

6. 显示成功信息和下一步指导

### 更新进度

**触发方式**：
- "更新进度 `<模块名>`"
- "更新 `<模块名>` 的学习进度"
- "同步进度 `<模块名>`"

**执行步骤**：

1. 读取模块的 `checklist.md`

2. 解析学习模式和完成项：
   - 检测已勾选的学习模式（快速/完整）
   - 统计已完成的 checkbox 数量
   - 根据学习模式计算对应的完成百分比

3. 读取 `PROGRESS.md` 获取当前进度

4. 生成更新预览：
   ```
   Progress: X% -> Y%
   Status: 进行中
   进度条: ██████░░░░░░░░░░░░░░ 30%
   ```

5. 用户确认后更新 `PROGRESS.md`：
   - 更新模块的完成百分比
   - 添加学习日志条目

**进度计算规则**：
- 快速模式：只统计"📚 快速模式"章节下的 checkbox
- 完整模式：统计"🛠️ 完整模式"章节下的所有 checkbox
- 未选择模式：基于完整模式计算

### 完成学习

**触发方式**：
- "完成学习 `<模块名>`"
- "标记 `<模块名>` 为已完成"
- "完成模块 `<模块名>`"

**执行步骤**：

1. 解析 `checklist.md` 计算完成百分比

2. 验证完成情况：
   - 如果完成度 < 100%，显示警告和未完成项列表
   - 询问是否确认完成
   - 如果用户坚持，可以强制完成（即使有未完成项）

3. 显示完成预览：
   - 最终完成百分比
   - 完成日期
   - 将要更新的文件

4. 用户确认后，更新以下文件：
   - **notes.md**：记录完成日期
   - **PROGRESS.md**：更新状态为"已完成"，进度设为 100%，添加完成日志

5. 重新计算总体进度和各优先级进度

### 同步学习计划

**触发方式**：
- "检查更新" 或 "查看是否有更新"
- "同步学习计划" 或 "更新学习计划"
- /study sync（仅检查）
- /study sync auto（执行同步）

**执行步骤**：

**模式 1：检查更新（check 模式，默认）**

1. 检查 upstream 远程仓库是否配置
   - 未配置：显示配置指引并退出

2. 从 CHANGELOG.md 提取当前版本和上游版本

3. 显示版本比较结果：
   - 如有更新：显示版本号差异和同步指引
   - 如已是最新：显示相应提示

**模式 2：执行同步（auto 模式）**

1. 检查更新（同 check 模式）

2. 如有更新，显示同步预览：
   - 当前版本和上游版本
   - 同步流程说明（备份、获取、合并、验证）
   - 个人数据保护说明

3. 用户确认后，调用 `scripts/sync.sh` 执行同步：
   - 备份个人数据到 `.backups/sync-<日期>/`
   - 从 upstream 获取最新更新
   - 合并更新到本地分支
   - 验证个人数据完整性

4. 显示同步结果和后续指引

**个人数据保护**：
- 以下文件已被 `.gitignore` 保护，不会被覆盖：
  - `PROGRESS.md`
  - `*/checklist.md`
  - `*/notes.md`
  - `.claude/LEARNING_BOOKMARKS.md`
  - `.claude/KNOWLEDGE_CACHE.md`

**非阻塞更新提醒**：
- 执行 `/study start`、`/study update`、`/study complete` 时
- 如检测到上游有更新，在输出前显示非阻塞提醒
- 提醒不打断命令执行，用户可稍后处理

## 学习书签命令

> 用于追踪学习过程中的疑问分支和深入学习点，确保探索后能返回主线。

### 创建书签

**触发方式**（任意一种）：
- "创建书签 `<名称>`"
- "添加书签 `<名称>`"
- "新建书签 `<名称>`"

**执行步骤**：

1. **自动获取上下文**：
   - 从 `PROGRESS.md` 获取当前"进行中"的模块
   - 从模块的 `checklist.md` 获取当前学习位置（最后一个有勾选的章节）

2. **询问用户**：
   - 疑问点描述（想要探索什么问题？）
   - 探索目标（可选，系统可智能生成）

3. **智能生成内容**：
   根据疑问点类型，自动生成探索目标和返回验证：

   | 疑问类型 | 探索目标 | 返回验证 |
   |----------|----------|----------|
   | 命令相关 | 阅读文档、实践使用、记录技巧 | 熟练使用该命令 |
   | 概念相关 | 阅读资料、整理笔记 | 能够解释概念 |
   | 配置相关 | 查看文档、实践配置 | 能够独立配置 |
   | 机制相关 | 阅读源码、实验验证 | 能够说明工作原理 |

4. **显示预览**：
   ```
   ╔═══════════════════════════════════════════════════════╗
   ║                    书签预览                           ║
   ╠═══════════════════════════════════════════════════════╣
   ║ 名称：深入理解 Tool Use 机制                          ║
   ║ 来源模块：claude-code-core                            ║
   ║ 主线位置：完整模式 > 第一部分 > 架构与概念            ║
   ╠═══════════════════════════════════════════════════════╣
   ║ 疑问点：                                              ║
   ║   Tool Use 的具体实现原理是什么？                      ║
   ╠═══════════════════════════════════════════════════════╣
   ║ 探索目标：                                            ║
   ║   - [ ] 阅读官方 Tool Use 文档                        ║
   ║   - [ ] 查看 Tool Use 的源码实现                      ║
   ╠═══════════════════════════════════════════════════════╣
   ║ 返回验证：                                            ║
   ║   - [ ] 能够解释 Tool Use 的工作原理                  ║
   ╚═══════════════════════════════════════════════════════╝
   ```

5. 用户确认后，将书签添加到 `LEARNING_BOOKMARKS.md`

### 书签探索行为规范

> 当书签状态为 🔵 探索中 时，AI 应遵循以下行为规范：

**探索阶段**：

1. **每次查询/操作完成后**：
   - 显示探索结果摘要（简洁，不超过 5 个要点）
   - ⭐ **关键：主动询问** "还有其他疑问吗？"
   - 等待用户响应，不要自动继续或返回主线

2. **用户回应的处理**：
   - 用户提出新疑问 → 继续探索，记录到探索笔记
   - 用户说"没有了"/"没问题" → 提示："可以说'完成书签'返回主线"
   - 用户说"完成书签" → 调用完成书签流程
   - 用户沉默/不明确 → 保持探索中状态，等待用户主动

3. **输出格式示例**：

   ```
   📚 探索结果：Tool Use 机制

   核心要点：
   • Tool Use 通过 function calling 实现
   • AI 主动调用外部工具，而非被动响应
   • 支持多工具并行调用

   ❓ 还有其他疑问吗？
      - 有疑问 → 直接告诉我
      - 没有了 → 说"完成书签"返回主线
   ```

**禁止行为**：

- ❌ 不要在查询完成后自动说"让我们返回主线学习"
- ❌ 不要推测用户没有疑问
- ❌ 不要在没有用户确认的情况下结束探索

**探索笔记记录**（可选）：

1. **触发条件**：用户明确说以下任一短语时触发
   - "记录一下"
   - "记到笔记里"
   - "保存这个"
   - "写到书签笔记"

2. **记录内容**：
   - 当前探索结果的要点摘要（与显示给用户的内容一致）
   - 记录时间戳

3. **记录位置**：
   - 添加到 `LEARNING_BOOKMARKS.md` 对应书签的"探索笔记"区域

4. **记录格式示例**：

   ```markdown
   ## 探索笔记

   ### 2026-02-28

   **Tool Use 机制探索**：

   - Tool Use 通过 function calling 实现
   - AI 主动调用外部工具，而非被动响应
   - 支持多工具并行调用
   ```

5. **记录后提示**：

   ```
   ✅ 已记录到书签笔记
   ```

6. **非自动记录**：
   - 用户未明确要求时，不自动记录
   - 避免文件频繁修改

## 注意事项

1. **用户确认**：修改多个文件前，总是先显示预览并等待用户确认
2. **日期格式**：统一使用 YYYY-MM-DD 格式
3. **文件路径**：所有路径相对于仓库根目录
4. **错误处理**：如果文件不存在或模块未找到，给出明确的错误提示
5. **进度计算**：基于实际勾选的 checkbox 数量，不要估算
6. **书签状态**：自动管理，用户通过命令交互，不手动编辑文件
7. **缓存策略**：每次询问用户，而非自动判断过期，给予用户完全控制
8. **内容获取优化**：拉取网页内容时使用 `retain_images=false` 跳过媒体文件，提升加载速度

**书签探索期间**：
- AI 应专注于书签疑问，避免讨论主线内容
- 探索结果使用要点列表，保持简洁（避免占用过多上下文）
- 每次查询后必须主动询问"还有其他疑问吗？"

**返回主线时**：
- 显示主线位置和下一步，帮助用户重新定位
- 避免延续书签话题，专注主线学习内容

## 理论知识展示行为规范

> 在学习过程中展示官方文档、概念解释、使用指南等理论知识时，AI 应遵循分块展示和用户确认机制，确保用户有充分时间理解内容。

### 适用范围

本规范适用于以下场景的展示：

| 场景 | 示例 |
|------|------|
| 官方文档 | GitHub README、API 文档、技术规范 |
| 概念解释 | 术语定义、原理说明、架构描述 |
| 使用指南 | 安装步骤、配置说明、操作流程 |
| 理论知识 | 学习资料、教程内容、最佳实践 |

### 分块展示策略

**触发条件**：

- 展示内容超过 15 行
- 内容包含多个独立的逻辑单元（如多个概念、多个步骤）

**分块原则**：

- 按逻辑单元分块（如按概念、按步骤、按小节）
- 每块不超过 15 行（不含确认提示）
- 优先在自然段落边界分块

**示例**：

```
第 1/3 块：核心概念
[展示前 15 行内容]

═══════════════════════════════════
❓ 是否继续下一部分？
   - 回复任意内容继续
   - "跳过"进入下一步
   - "暂停"稍后继续
═══════════════════════════════════

[用户确认后...]

第 2/3 块：实现原理
[展示接下来的内容]
```

### 确认提示格式

每块理论内容展示后，使用以下格式询问用户：

```
═══════════════════════════════════
❓ 是否继续下一部分？
   - 回复任意内容继续
   - "跳过"进入下一步
   - "暂停"稍后继续
═══════════════════════════════════
```

### 用户响应处理

| 用户输入 | AI 行为 |
|---------|---------|
| 任意非指令内容（如"继续"、"好的"、"y"） | 继续展示下一块 |
| "跳过" | 跳过剩余理论内容，直接进入下一步操作 |
| "暂停" | 停止展示，告知用户可回复"继续"恢复 |
| "完成" | 结束理论展示，进入下一步操作 |

### 特殊场景处理

**短篇内容**（15 行以内）：

- 直接完整展示
- 仍需等待用户确认后再继续下一步
- 确认提示可简化为："请阅读完成后回复任意内容继续"

**用户说"暂停"后**：

```
═══════════════════════════════════
⏸️ 已暂停理论展示
   回复"继续"可恢复阅读
═══════════════════════════════════
```

**用户说"跳过"后**：

```
═══════════════════════════════════
⏭️ 已跳过剩余理论内容
   正在进入下一步...
═══════════════════════════════════
```

### 来源标注规范

> 当展示有明确来源的内容时，应标注信息来源，方便用户对比验证。

**需要标注的场景**：

| 内容类型 | 标注方式 | 示例 |
|----------|----------|------|
| 官方文档 | 文档名称 + 链接 | `来源：Claude Code 官方文档 https://code.claude.com/docs` |
| GitHub README | 仓库名 + 文件路径 | `来源：anthropics/claude-code/blob/main/README.md` |
| API 文档 | 文档标题 + URL | `来源：MCP SDK API Reference` |
| 缓存知识 | 缓存日期 | `来源：知识缓存 (2026-03-01)` |
| AI 生成内容 | 无需标注 | （直接展示） |

**标注格式**：

在内容块之前添加来源说明：

```
📚 来源：Claude Code 官方文档
https://code.claude.com/docs/getting-started

───────

[内容开始]
```

或使用简洁格式（适合内容较短时）：

```
📚 [Claude Code 官方文档]

[内容开始]
```

**无需标注的场景**：

- AI 自己解释或总结的内容
- 用户的提问或输入
- 没有明确来源的通用知识

### 禁止行为

- ❌ 不要在展示完理论内容后自动继续下一步
- ❌ 不要推测用户已经读完内容
- ❌ 不要在用户未确认的情况下进入下一步
- ❌ 不要一次性展示超过 15 行的理论内容
- ❌ 不要因为 `accept edits on` 模式开启而省略理论确认

### 优先级说明

当多个行为规范同时适用时，按以下优先级执行：

1. **用户明确指令**（如"直接展示"、"不要分段"）
2. **书签探索规范**（当书签状态为 🔵 探索中）
3. **理论知识展示规范**（本规范）

例如：
- 书签探索期间的展示，优先遵循书签探索规范
- 用户说"一次性展示完所有内容"，则直接完整展示

### 独立于 `accept edits on` 设置

> 重要说明：理论知识确认机制独立于文件编辑设置

- `accept edits on`：仅影响文件编辑操作，不影响理论展示
- 即使 `accept edits on` 开启，展示理论内容时仍需等待用户确认
